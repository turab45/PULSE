# ClinicAgent

ClinicAgent is a Python prototype for clinical documentation support. It takes a
raw consultation transcript and runs it through a supervised multi-agent
workflow that produces:

1. A structured SOAP note
2. A care-gap review
3. A follow-up plan
4. A final clinician-review report

The project uses LangGraph for workflow orchestration, LangChain message and
prompt primitives, and Groq-hosted Llama models for content generation.

> Important: this project is not a medical device and is not ready for
> unsupervised clinical use. It is a production-shaped research baseline: the
> code is modular, testable, configurable, and safer than the original
> prototype, but every output must still be reviewed and approved by a licensed
> clinician before use.

## What The System Does

ClinicAgent models a clinical documentation workflow as a graph:

```text
transcript
   |
   v
supervisor -> scribe -> supervisor -> gap_detector
                                      |
                                      v
                              supervisor -> planner
                                      |
                                      v
                              supervisor -> reviewer -> supervisor -> end
```

Each content agent has one job:

- `scribe`: converts the consultation transcript into a SOAP note.
- `gap_detector`: identifies missed follow-up needs, quality gaps, safety
  concerns, or guideline considerations for clinician review.
- `planner`: drafts follow-up actions and a patient-friendly message.
- `reviewer`: checks the generated documentation for accuracy, completeness,
  clarity, and hallucinated clinical details.
- `supervisor`: deterministically routes the workflow. It does not use an LLM,
  which makes routing testable and prevents model-driven loops.

## Setup

Create and activate a virtual environment:

```bash
python3 -m venv .venv
source .venv/bin/activate
```

Install dependencies:

```bash
pip install -r requirements.txt
```

Create a local environment file:

```bash
cp .env.example .env
```

Edit `.env` and add your Groq API key:

```bash
GROQ_API_KEY=your_groq_api_key_here
```

Optional model settings:

```bash
CLINIC_AGENT_MODEL=llama-3.3-70b-versatile
CLINIC_AGENT_TEMPERATURE=0.1
CLINIC_AGENT_MAX_TOKENS=2048
```

## Run The Demo

Run with the built-in sample transcript:

```bash
python3 -m src.main
```

Run with your own transcript:

```bash
python3 -m src.main --transcript-file path/to/transcript.txt
```

You can also keep the old command:

```bash
python3 healthcare.py
```

## Run Tests

The core parser and supervisor tests do not require a Groq key:

```bash
python3 -m unittest discover -s tests
```

Compile-check the package:

```bash
python3 -m compileall src tests healthcare.py
```

## Workflow State

The shared graph state is defined in `src/state.py`.

Important fields:

- `messages`: LangChain message history, merged by LangGraph's message reducer.
- `transcript`: the source consultation transcript.
- `soap_note`: the structured note generated by the scribe.
- `care_gaps`: plain list of gap strings. An empty list means gap detection has
  not run yet. `["No gaps identified"]` means it ran and found no structured
  gaps.
- `follow_up_plan`: care-coordination output.
- `final_report`: reviewer output.
- `review_status`: `approved` or `issues`.
- `next`: supervisor-selected next step.

## Why The Supervisor Is Deterministic

The first prototype used an LLM prompt to decide where the graph should go next.
That is fragile in production because a routing typo can create loops or skip
required review steps.

The current supervisor uses simple state rules:

1. No SOAP note: run `scribe`
2. SOAP exists but gap analysis has not run: run `gap_detector`
3. Gap analysis exists but plan is missing: run `planner`
4. Plan exists but final report is missing: run `reviewer`
5. Final report exists: end

This keeps workflow control observable, unit-testable, and cheap.

## Safety Boundaries

ClinicAgent is designed as clinician-support software, not an autonomous care
system. To move toward real deployment, keep these boundaries:

- Human review is required before sending patient messages or adding content to
  an EHR.
- Generated text must not be treated as a diagnosis, prescription, or final
  clinical order.
- The prompts explicitly forbid invented facts, labs, medications, or decisions,
  but prompts alone are not sufficient clinical safety controls.
- The system should be evaluated against representative transcripts before any
  operational use.
- Patient data must be handled according to local privacy and security rules.
  Do not send protected health information to a model provider unless the
  provider, contract, deployment, and jurisdiction make that permissible.

## Production Readiness Checklist

Completed in this version:

- Modular package layout under `src/`
- Lazy LLM creation so imports and tests do not require credentials
- Environment-based configuration
- Deterministic graph routing
- Correct LangGraph conditional edges
- Reviewer writes to `final_report` instead of corrupting `care_gaps`
- Fail-fast validation for required SOAP and follow-up-plan sections
- Parser and supervisor unit tests
- `.env.example` for local configuration
- Backward-compatible `healthcare.py` wrapper
- README with setup, architecture, usage, testing, and safety notes

Still needed before real clinical deployment:

- Formal clinical validation with clinicians and representative cases
- PHI-safe deployment design and audit logging
- Persistent checkpoint/storage backend instead of in-memory checkpoints
- Structured outputs with schemas rather than free-text parsing
- Integration tests that mock the LLM layer
- Monitoring for model failures, latency, and unsafe outputs
- EHR integration review and approval workflow
- Security review, dependency scanning, and CI/CD

## Development Notes

Use `python3 -m src.main` as the canonical command. This ensures package imports
resolve consistently.

The LLM is created in `src/config.py` only when an agent actually runs. That
means parser and supervisor tests stay fast and do not need network access.

When changing prompts, update tests if the parser contract changes. If you need
more reliable production outputs, prefer structured output schemas over adding
more parsing rules to free text.
